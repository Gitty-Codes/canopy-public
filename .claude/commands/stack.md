Read and display `skills/canopy-stack.md` in full.

After displaying it, ask: "Does anything need to be updated?"

If the founder says yes, work with them to update the file directly.
Keep the same format and level of specificity — this file is situational truth,
not a capabilities list. It describes what actually exists and what has been decided.

Things that belong in the stack file:
- What is built and working
- What has been tried and why it didn't work
- Key architectural decisions and their reasoning
- What is currently in flight
- The public API of the harness (modes, inputs, outputs)

Things that do not belong:
- Aspirational features not yet built
- Generic descriptions of what agents do (that's the constitution's job)
- Tutorial content
- Anything that would be stale within a week

The Challenger applies here: if something in the stack file is no longer true,
say so and remove it rather than letting stale facts accumulate.
