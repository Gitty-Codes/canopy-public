# Org Memory — Organizational Context Schema

Org memory is the difference between Claude knowing who an organization is and
Claude treating them as a generic nonprofit. Every other skill in this library
becomes more precise when org memory is loaded. It is the foundation.

---

## What org memory contains

An org memory document has eight sections. All are useful. None are required to
start — fill in what is known, mark gaps explicitly, improve over time.

**IDENTITY** — Official name, short name, location, founding year, model, mission.
The mission should be in the organization's own words, not paraphrased.

**PROGRAM** — What students/participants do, how often, demographics, cost to
participants. Specifics over generalities. "Free. Zero." is better than "low-cost."

**OUTCOMES** — The metric that stops readers. Graduation rates, post-secondary rates,
named alumni stories (with permission). If the org tracks attendance or academic
performance data, include it. Outcomes are funding assets.

**PARTNERSHIPS** — Each active partnership named with: nature of relationship,
duration, who holds the relationship at the org, what the partner has said publicly
(if anything). Partnerships are credibility signals and warm introduction pathways.

**FUNDING** — Annual budget, funding breakdown (grants / earned / individual),
active funder relationships (who, how much, who holds the relationship), funders
who have declined or lapsed. Never approach a funder who already said no without
a strategy for why now is different.

**VOICE AND BRAND** — Three words that feel true. Words or framings they avoid.
How they want to talk about sensitive partnerships (e.g. tribal relationships as
trust, not demographic credential). Brand materials locations.

**STAFF AND BOARD** — Who does what. Who holds which external relationships.
Honest board capacity assessment. Board matching gift potential.

**WHAT THEY WANT** — In their own words: what would "not worrying about money"
look like? What do they want to do more of? What relationships are they most
excited to build? What has surprised them about what has worked?

---

## Loading protocol

When org memory is provided, apply it as follows:

1. **Lead with their outcomes, not their story.** The strongest outcome metric
   comes before the founding year. Funders and donors make decisions in the first
   paragraph.

2. **Use their language, not generic nonprofit language.** If their voice doc says
   they avoid "at-risk youth," that phrase never appears in anything you write for them.

3. **Treat partnerships as assets.** A tribal partnership that signals trust earned
   over years is not a demographic footnote. A Seattle Symphony relationship is a
   warm introduction to a donor pool the org cannot access on their own.

4. **Name the gaps.** If a section is missing, note it: "No alumni stories loaded —
   add to org memory before grant deadline." Missing context produces generic output.

5. **Do not invent.** If the budget is unknown, leave it unknown. If the board
   capacity is "we honestly don't know," that is the answer. Invented specifics
   contaminate every output that uses them.

---

## When org memory is incomplete

Partial org memory is better than no org memory. What is known shapes the output
even when much is still unknown. But name the gaps so the team knows what to fill in.

The org-memory-intake.md document in a client's project folder is the source of truth.
Always check for an existing intake before asking the organization to re-explain themselves.

---

## Updating org memory

Org memory should be updated when:
- A significant funder relationship begins or ends
- A new outcome metric becomes available (new grad data, attendance stats)
- A staff or board change affects relationship ownership
- A new partnership forms or an existing one deepens
- The organization's voice or framing evolves

The org memory document is a living record. Stale org memory is worse than no org
memory — it produces confident outputs that are factually wrong.
